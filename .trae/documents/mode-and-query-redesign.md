# 模式切换 & 查询参数区域 — 设计分析 & 优化方案

## 一、现状问题诊断

### 问题 1：模式切换重复出现在两个位置 ❌

**事实**：
- [DataTableHeader.tsx](file:///d:/std/postman-app/src/components/data-table/DataTableHeader.tsx#L44-L58) 里有一个完整的 `dt-mode-selector`（Simple / Detailed 双标签分段控件）
- [DataTableToolbar.tsx](file:///d:/std/postman-app/src/components/data-table/DataTableToolbar.tsx#L120-L127) 里又有一个 `dt-toolbar-mode-toggle`（S / D 单按钮循环切换）

**问题**：
1. 两个按钮做同一件事，用户困惑——到底用哪个？
2. 工具栏的 S/D 按钮看不到完整模式名，需要 tooltip，认知负担高
3. 操作逻辑不一致——Header 是选择模式，Toolbar 是 Toggle，但 `window.confirm` 都出现两次
4. 纯属冗余，增加维护成本和视觉噪音

### 问题 2：查询参数折叠设计反直觉 ❌

**事实**：
- 数据加载前：展开状态显示 Project ID + Test Run ID + Search 按钮
- 数据加载后：可以通过 "Query Params" 按钮折叠参数区域
- 折叠后：只显示 "Expand" 一行，数据表格空间变大

**问题**：
1. **语义混乱**：按钮写 "Query Params" / "Expand"，用户不理解这是什么操作
2. **非标准交互**：查询参数是"一次性"操作——查完就应该最小化，但现在的折叠只是隐藏了 input 但保留了一大块空白区域
3. **空间浪费**：折叠后仍然占用一整行高度（`collapsed` 状态 padding 仍然在）
4. **缺少上下文**：折叠后用户看不到当前的 Project ID / Test Run ID 是什么
5. **查询历史缺失**：没有「最近查询」快捷入口，每次都要重新输入

### 问题 3：查询参数与模式切换的关系不清晰 ❌

两个 mode 对应不同的数据模型（Simple 列少、Detailed 列多），但目前：
- 切换模式时没有任何关于"数据变化"的预览/说明
- 用户不知道切换后表格会多出/少出哪些列
- `window.confirm` 提示太粗暴

## 二、优秀的设计方案

### 方案 A：统一查询栏（推荐 ✅）

**核心思路**：将模式选择和查询融合为一个紧凑的「查询栏」，扁平化信息层级。

```
┌──────────────────────────────────────────────────────────┐
│  🏷 Simple │ Detailed    🔍  Project ID  │  Test Run ID  │  [Search]  │  📋 历史  ▼ │
└──────────────────────────────────────────────────────────┘
                                                   ↓ 查到数据后自动收起为一行
┌──────────────────────────────────────────────────────────┐
│  🏷 Simple │ Detailed    📌 ScenarioTesting / SYSQT-...   │  [🔄刷新] [✏️修改] │
└──────────────────────────────────────────────────────────┘
```

**设计要点**：

1. **模式选择器保留在左侧**，但改为 **Icon + 缩写** 紧凑风格（减少视觉重量）
   - Simple: `⚡ S` 
   - Detailed: `📋 D` 
   - 带 tooltip 显示完整描述："Simple — 仅显示核心字段" / "Detailed — 显示完整执行详情"

2. **查询参数区三态切换**（而非布尔折叠）：
   - **初始态**：完整展示两个输入框 + Search 按钮（首次进入/未查询过）
   - **运行态**：输入框切换为只读标签，显示 `📌 Project/TestRun` 路径，右侧出现 `[🔄 刷新] [✏️ 修改]` 两个按钮
   - **编辑态**：点击修改回到输入框模式

3. **查询历史下拉**（在 Search 按钮右侧）：
   - 下拉显示最近 5 条查询记录
   - 每条记录格式：`Project ID → Test Run ID` + 时间戳
   - 点击直接切换

4. **模式切换预览**：
   - 切换模式前在 mode tab 上 hover 时显示一个小 popover
   - 列出当前模式 vs 目标模式的列数差异
   - 例如：「切换到 Detailed 模式将新增 11 列：duration, executedTime, ...」

---

### 方案 B：沉浸式搜索面板 + 持久化查询栏

**核心思路**：空状态时以「搜索面板」为主视觉，查到数据后查询栏自动缩为迷你状态栏。

```
  初始态（无数据）                          数据态（有数据）
┌──────────────────────┐        ┌─────────────────────────────────────┐
│                      │        │  S D │ 📌 ScenarioTesting / SYS...  │
│   🎯 搜索测试数据     │        │  [列] [视图] [导出▼] [密度▦]       │
│                      │        ├─────────────────────────────────────┤
│  Project ID    [___] │   →    │  (表格数据区域)                     │
│  Test Run ID   [___] │        │                                     │
│                      │        │                                     │
│     [ Search ]       │        └─────────────────────────────────────┘
│                      │
│  最近查询：           │
│  · ProjA / Run001    │
│  · ProjB / Run002    │
│                      │
└──────────────────────┘
```

**设计要点**：

1. **空状态时**：查询区作为 Hero 视觉中心（带插画 + 大输入框）
2. **查询后**：查询栏缩为一行迷你状态条，融入工具栏
3. **模式切换**：合并到工具栏密度按钮旁，占据一个按钮位

## 三、推荐方案详解（方案 A 细化）

我认为**方案 A**是最佳选择，原因：
- 改动最小，不破坏现有布局
- 解决了「两处模式切换」的冗余
- 查询参数的三态设计既节省空间又有上下文
- 查询历史大幅提升重复查询的效率

### 实施步骤

#### 步骤 1：删除 Toolbar 中的模式切换
- 移除 `DataTableToolbar.tsx` 中的 `dt-toolbar-mode-toggle` 按钮（第 120-127 行）
- 移除 toolbar 中 `handleModeToggle` 和 `DataTableMode` import
- 模式切换只保留在 Header 中

#### 步骤 2：重构 Header 查询区为三态
- 新增状态变量：`queryState: 'idle' | 'editing' | 'sealed'`
  - `idle`：无数据时显示输入框
  - `editing`：有数据但用户在修改参数
  - `sealed`：有数据时显示只读标签
- `sealed` 态 UI：
  ```
  📌 ScenarioTesting / SYSQT-20260317-1052   [🔄] [✏️]
  ```
- 点击 [✏️] 进入 `editing` 态 → 显示输入框 + Search 按钮
- Search 完成后自动回到 `sealed` 态

#### 步骤 3：添加查询历史
- 从 store 读取 `queryHistory`（已存在）
- 在 Search 按钮右侧增加 `📋` 按钮
- 点击弹出下拉列表（最近 5 条）
- 点击历史项自动填入并搜索

#### 步骤 4：模式切换优化
- Header 模式切换增加 hover tooltip 说明列差异
- Header 模式切换改用 `setMode`（已有），但不再用 `window.confirm`
- 添加一个简洁的确认提示（列差异预览）

#### 步骤 5：移除 `queryCollapsed` 机制
- 删除展开/折叠按钮
- 改用三态自动管理空间
