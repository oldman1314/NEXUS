# API Studio 新功能建议计划

基于对现有代码库的全面分析，以下是按优先级和类别组织的新功能建议。

***

## 一、协议扩展类（高优先级）

### 1. WebSocket 客户端

* **现状**：当前仅支持 HTTP 协议请求

* **建议**：新增 WebSocket 调试能力

* **功能点**：

  * WebSocket 连接建立与断开

  * 实时消息发送/接收（支持 Text 和 Binary）

  * 消息历史记录与搜索

  * 心跳/Ping-Pong 监控

  * 连接状态指示器

* **实现思路**：在 RequestView 旁新增 WebSocketView，复用现有的集合管理和环境变量系统

### 2. GraphQL 客户端

* **现状**：只能通过 JSON Body 手动编写 GraphQL 查询

* **建议**：内置 GraphQL 专用编辑器

* **功能点**：

  * GraphQL 语法高亮与自动补全

  * Schema Explorer（从 Introspection 自动加载）

  * Query/Mutation/Subscription 分标签

  * Variables 面板（JSON 编辑器）

  * 请求历史按操作名分组

* **实现思路**：在 Body 标签页中新增 `graphql` 类型，使用 graphql-language-service 提供补全

### 3. SSE (Server-Sent Events) 客户端

* **现状**：不支持流式响应调试

* **建议**：支持 SSE 协议调试

* **功能点**：

  * 实时事件流展示

  * 事件类型过滤

  * Last-Event-ID 支持

  * 连接状态监控

***

## 二、Mock 与测试类（高优先级）

### 4. 本地 Mock Server

* **现状**：没有 Mock 能力，前端开发必须依赖真实后端

* **建议**：内置 Mock 服务器

* **功能点**：

  * 从集合/请求自动生成 Mock 规则

  * 自定义 Mock 响应（状态码、Headers、Body、延迟）

  * 支持路径参数和查询参数匹配

  * Mock 规则优先级管理

  * 本地端口启动 Mock 服务（通过 Electron 主进程）

* **实现思路**：在 Electron 主进程中启动 Express/Fastify 服务器，渲染进程通过 IPC 管理 Mock 规则

### 5. 自动化测试套件

* **现状**：仅有单请求级别的 Tests 脚本

* **建议**：支持测试套件级别的组织和执行

* **功能点**：

  * 将集合组织为测试套件（Collection Runner）

  * 按顺序/并发执行集合中的请求

  * 测试结果汇总（通过率、失败详情、耗时统计）

  * 测试报告导出（HTML/PDF/JSON）

  * 延迟设置（请求间隔）

  * 失败时停止/继续选项

  * 迭代运行（支持数据驱动测试，CSV/JSON 数据文件）

* **实现思路**：新增 CollectionRunnerView，复用现有的 script-engine 和 request 发送逻辑

### 6. API 监控（Scheduled Requests）

* **现状**：只能手动发送请求

* **建议**：支持定时自动发送请求并监控响应

* **功能点**：

  * 定时任务配置（Cron 表达式或简单间隔）

  * 响应状态码/耗时/Body 断言

  * 异常告警（状态码变化、超时、断言失败）

  * 监控历史趋势图

  * 后台运行（系统托盘）

***

## 三、数据可视化类（中优先级）

### 7. 测试结果仪表盘

* **现状**：Data Table 仅有表格展示，缺少可视化统计

* **建议**：新增图表统计仪表盘

* **功能点**：

  * 测试结果分布饼图（passed/failed/blocked 等）

  * 执行耗时趋势折线图

  * 按优先级/环境/模块的分组统计柱状图

  * 缺陷关联热力图

  * 通过率趋势

  * 可导出为图片

* **实现思路**：引入 Recharts 或 ECharts（轻量级），新增 DashboardView

### 8. 请求性能分析

* **现状**：仅显示总耗时

* **建议**：详细的请求性能分解

* **功能点**：

  * DNS 解析耗时

  * TCP 连接耗时

  * TLS 握手耗时

  * TTFB (Time To First Byte)

  * 内容下载耗时

  * 瀑布图（Waterfall）展示

  * 与历史请求对比

***

## 四、协作与分享类（中优先级）

### 9. 请求分享（可导入链接）

* **现状**：只能通过导出 JSON 文件分享

* **建议**：生成可分享的请求链接/二维码

* **功能点**：

  * 将请求配置编码为 URL 参数或短链接

  * 生成二维码供移动端扫描

  * 一键复制分享链接

  * 打开分享链接自动导入请求

* **实现思路**：将请求配置 Base64 编码后附加到 URL，或使用本地 HTTP 服务提供短链接

### 10. API 文档生成

* **现状**：没有文档生成能力

* **建议**：从集合自动生成 API 文档

* **功能点**：

  * 从请求配置和脚本注释提取文档信息

  * 支持手动补充描述、示例

  * 生成 Markdown / HTML 格式文档

  * 在线预览

  * 支持按集合/文件夹组织

* **实现思路**：新增 DocGeneratorView，解析集合数据生成 Markdown，使用 marked 渲染预览

***

## 五、远程工具增强类（中优先级）

### 11. SFTP 文件管理器

* **现状**：SSH 终端仅支持命令行操作

* **建议**：在 SSH 连接上增加图形化文件管理

* **功能点**：

  * 双栏文件浏览器（本地 + 远程）

  * 拖拽上传/下载

  * 文件编辑（远程文件在线编辑）

  * 权限修改

  * 目录书签

* **实现思路**：在 Electron 主进程的 ssh2 连接上增加 SFTP 操作，渲染进程通过 IPC 调用

### 12. 数据库客户端

* **现状**：无数据库连接能力

* **建议**：集成轻量级数据库客户端

* **功能点**：

  * 支持 MySQL / PostgreSQL / SQLite / Redis

  * SQL 编辑器（语法高亮、自动补全）

  * 查询结果表格展示

  * 连接管理（保存/切换连接配置）

  * 数据导入/导出

* **实现思路**：在 Electron 主进程中集成数据库驱动，渲染进程通过 IPC 通信

***

## 六、开发者体验类（低优先级）

### 13. AI 辅助功能

* **现状**：无 AI 集成

* **建议**：集成 AI 辅助 API 开发

* **功能点**：

  * 自然语言生成请求（"发送一个获取用户列表的 GET 请求"）

  * AI 生成测试脚本

  * AI 分析响应数据

  * AI 生成 Mock 数据

  * API 文档自动补全描述

  * 错误诊断与修复建议

* **实现思路**：集成 OpenAI/Claude API，用户配置自己的 API Key，通过 prompt engineering 实现

### 14. 插件系统

* **现状**：功能全部内置，无法扩展

* **建议**：支持第三方插件扩展

* **功能点**：

  * 插件市场/管理器

  * 插件 API（请求拦截、响应处理、UI 扩展）

  * 沙箱执行环境

  * 插件配置面板

* **实现思路**：定义插件接口规范，使用 Web Worker 或 iframe 沙箱执行插件代码

### 15. 请求链（Request Chaining）

* **现状**：只能通过工作流实现请求间数据传递

* **建议**：在集合级别支持简单的请求链

* **功能点**：

  * 从前一个请求的响应中提取变量

  * 自动传递到后续请求

  * 可视化依赖关系

  * 比工作流更轻量的编排方式

* **实现思路**：在环境变量中增加"动态变量"类型，支持 `{{response.body.id}}` 引用

### 16. 键盘快捷键自定义

* **现状**：快捷键硬编码

* **建议**：允许用户自定义快捷键

* **功能点**：

  * 快捷键配置面板

  * 冲突检测

  * 预设方案（Vim / Emacs / Default）

  * 导入/导出快捷键配置

***

## 七、工作流增强类（低优先级）

### 17. 工作流子流程节点

* **现状**：工作流不支持嵌套

* **建议**：支持子流程节点引用其他工作流

* **功能点**：

  * 子流程节点类型

  * 工作流引用与嵌套

  * 参数传递（输入/输出映射）

  * 递归防护

### 18. 工作流导出为代码

* **现状**：工作流只能在应用内执行

* **建议**：将工作流导出为可独立运行的脚本

* **功能点**：

  * 导出为 Node.js 脚本

  * 导出为 Python 脚本

  * 导出为 Shell 脚本（cURL 链式调用）

  * 包含环境变量配置模板

***

## 推荐实施路线

| 阶段          | 功能                     | 理由                             |
| ----------- | ---------------------- | ------------------------------ |
| **Phase 1** | WebSocket 客户端、自动化测试套件  | 补全协议支持是刚需；测试套件是 Postman 核心功能缺失 |
| **Phase 2** | Mock Server、测试结果仪表盘    | Mock 大幅提升前端开发效率；仪表盘让数据表价值最大化   |
| **Phase 3** | GraphQL 客户端、SFTP 文件管理器 | GraphQL 是主流趋势；SFTP 补全远程工具能力    |
| **Phase 4** | API 文档生成、请求分享          | 提升团队协作效率                       |
| **Phase 5** | AI 辅助、插件系统             | 锦上添花，提升差异化竞争力                  |
